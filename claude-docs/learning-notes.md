## Anthropic 구조화 출력 — SDK 없이도 프롬프트↔파서 에코 문제를 원천 제거

**한 줄 요약**: LLM에게 "JSON만 내라"고 프롬프트로 부탁하는 대신 Messages API의
`output_config.format`(json_schema)을 쓰면 **응답이 스키마 준수 JSON임이 API 레벨에서
보장**된다 — 일반 JSON body 필드라 anthropic SDK 없이 stdlib HTTP로도 쓸 수 있다.

**자세한 설명**:

- **문제의 구조**: 자유형식 텍스트에서 JSON을 뽑는 파서는 ① 서문/코드펜스/후문 산문,
  ② 프롬프트가 담은 형식 예시의 에코, ③ 절단(max_tokens)과 싸워야 한다. 이 프로젝트의
  감지 어댑터 파서가 그 휴리스틱으로 적대적 리뷰 3라운드를 거치며 계속 뚫렸다(T-031→T-032).
  프롬프트 레벨 지시("JSON만 출력")는 **부탁**이지 **계약**이 아니다.
- **구조화 출력은 싸움 자체를 없앤다**: 요청 body에
  `"output_config": {"format": {"type": "json_schema", "schema": {...}}}` 를 넣으면 응답
  text 블록이 스키마 유효 JSON만으로 온다 — 파서의 에코 억제·컨테이너 선택 휴리스틱
  대부분이 `json.loads` + shape 검사로 축소된다.
- **핵심 통찰**: "SDK 기능"처럼 보이는 것 상당수가 실은 그냥 **HTTP body 필드**다. 이
  프로젝트의 '런타임 의존성 0'(stdlib urllib) 제약과 전혀 충돌하지 않는다 — 비유하면 SDK는
  택배 기사일 뿐이고, 상자에 뭘 넣을 수 있는지(body 필드)는 우체국(API)이 정한다.
- **한계/주의**: 절단은 스키마 보장과 별개로 여전히 가능 → `stop_reason` 검사는 그대로
  필요. 어시스턴트 프리필 방식은 opus-4-8 계열에서 400이라 대안이 아님.
- **면접 Q&A 대비** — "LLM 출력 파싱을 어떻게 견고하게 하나?":
  ① 가능하면 구조화 출력으로 계약을 프롬프트 레벨 → API 레벨로 올린다.
  ② 불가하면 '첫 파싱값'이 아니라 판별 키 기반 의미 선택 + **극단 카디널리티(0건·1건·
  동수·절단) 테스트**로 휴리스틱을 검증한다.
  ③ '빈 유효 결과 vs 추출 실패'를 구분해 모호하면 fail-loud(무성 0건 = 벤치 오염).

**코드 위치**: `benchmarks/detection/detect_bench/detect.py` — `ClaudeDetectorPort`(stdlib
urllib로 body 직조립; 실측 단계에서 output_config 적용 검토), `parse_detection_response`
(현행 의미 기반 파서). PR #11 3R 리뷰에서 도출.

**관련 노트**: `troubleshooting.md` T-031(LLM 출력 구조 추출은 위치가 아니라 의미로) ·
T-032(프롬프트 예시는 비파싱 표기로 + 극단 카디널리티 테스트).
