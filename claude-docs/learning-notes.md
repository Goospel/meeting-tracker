## Anthropic 구조화 출력 — SDK 없이도 프롬프트↔파서 에코 문제를 원천 제거

**한 줄 요약**: LLM에게 "JSON만 내라"고 프롬프트로 부탁하는 대신 Messages API의
`output_config.format`(json_schema)을 쓰면 **응답이 스키마 준수 JSON임이 API 레벨에서
보장**된다 — 일반 JSON body 필드라 anthropic SDK 없이 stdlib HTTP로도 쓸 수 있다.

**자세한 설명**:

- **문제의 구조**: 자유형식 텍스트에서 JSON을 뽑는 파서는 ① 서문/코드펜스/후문 산문,
  ② 프롬프트가 담은 형식 예시의 에코, ③ 절단(max_tokens)과 싸워야 한다. 이 프로젝트의
  감지 어댑터 파서가 그 휴리스틱으로 적대적 리뷰 3라운드를 거치며 계속 뚫렸다(T-031→T-032).
  프롬프트 레벨 지시("JSON만 출력")는 **부탁**이지 **계약**이 아니다.
- **구조화 출력은 싸움 자체를 없앤다**: 요청 body에
  `"output_config": {"format": {"type": "json_schema", "schema": {...}}}` 를 넣으면 응답
  text 블록이 스키마 유효 JSON만으로 온다 — 파서의 에코 억제·컨테이너 선택 휴리스틱
  대부분이 `json.loads` + shape 검사로 축소된다.
- **핵심 통찰**: "SDK 기능"처럼 보이는 것 상당수가 실은 그냥 **HTTP body 필드**다. 이
  프로젝트의 '런타임 의존성 0'(stdlib urllib) 제약과 전혀 충돌하지 않는다 — 비유하면 SDK는
  택배 기사일 뿐이고, 상자에 뭘 넣을 수 있는지(body 필드)는 우체국(API)이 정한다.
- **한계/주의**: 절단은 스키마 보장과 별개로 여전히 가능 → `stop_reason` 검사는 그대로
  필요. 어시스턴트 프리필 방식은 opus-4-8 계열에서 400이라 대안이 아님.
- **면접 Q&A 대비** — "LLM 출력 파싱을 어떻게 견고하게 하나?":
  ① 가능하면 구조화 출력으로 계약을 프롬프트 레벨 → API 레벨로 올린다.
  ② 불가하면 '첫 파싱값'이 아니라 판별 키 기반 의미 선택 + **극단 카디널리티(0건·1건·
  동수·절단) 테스트**로 휴리스틱을 검증한다.
  ③ '빈 유효 결과 vs 추출 실패'를 구분해 모호하면 fail-loud(무성 0건 = 벤치 오염).

**코드 위치**: `benchmarks/detection/detect_bench/detect.py` — `ClaudeDetectorPort`(stdlib
urllib로 body 직조립; 실측 단계에서 output_config 적용 검토), `parse_detection_response`
(현행 의미 기반 파서). PR #11 3R 리뷰에서 도출.

**관련 노트**: `troubleshooting.md` T-031(LLM 출력 구조 추출은 위치가 아니라 의미로) ·
T-032(프롬프트 예시는 비파싱 표기로 + 극단 카디널리티 테스트).

## 정밀도·재현율 비대칭과 '실패 방향' — 실 감지기는 오탐보다 겹치는 경계 과소감지로 샌다

**한 줄 요약**: 감지 제품에서 정밀도(오탐 안 냄)와 재현율(안 놓침)은 **대칭이 아니다** —
오탐은 사용자 신뢰를 즉시 깨는 지뢰, 미탐은 조용히 가치가 새는 구멍. 첫 실측(Opus 4.8, 골든
3건)에서 실 감지기는 **정밀도 1.00 · 재현율 0.87**로 실패가 미탐 쪽에 몰렸고, 미탐은 전부
**한 발언이 두 라벨에 걸치는 겹침**에서 상위 서사가 개별 라벨을 흡수한 케이스였다.

**자세한 설명**:

- **왜 실패 방향을 봐야 하나**: 단일 F1만 보면 "0.93, 괜찮네"로 끝난다. 하지만 **어느 방향으로
  틀리는가**가 제품 결정을 바꾼다. 이 감지기는 정밀도가 완벽(찍은 13건 전부 진짜) — 즉
  사용자에게 "이거 모순이에요" 알림이 뜨면 **틀린 적이 없다**. 대신 미묘한 겹침 단절을 놓친다.
  이 실패 방향이면 제품은 "놓칠 순 있어도 헛알림으로 신뢰를 깨진 않는다"는 포지션을 잡을 수 있다.
  반대 방향(재현율 높고 정밀도 낮음)이었다면 "지뢰밭"이라 알림 자체를 못 켠다. **F1이 같아도
  두 감지기는 전혀 다른 제품이다.**
- **겹치는 라벨에서 '흡수'가 일어난다**: 놓친 2건 중 하나는 p4가 "무상연장 반대→찬성"으로
  자기모순(모순)인데, 그 두 번째 발언이 동시에 **팀 결정이 뒤집힌 번복**의 근거이기도 했다.
  LLM은 **더 큰 서사(번복)를 택하고 그 안에 개별 자기모순을 흡수**해, 모순을 별도로 안 찍었다.
  한 근거가 여러 라벨에 걸칠 때 모델은 하나의 상위 프레임으로 뭉개는 경향 — 라벨이 상호배타가
  아닌 태스크(모순⊂번복 가능)의 구조적 난점이다.
- **객체탐지식 채점이 이걸 드러낸다**: type-strict 매칭만 보면 "놓쳤다"로 끝나지만, 이 하네스는
  **type-무관 localization**을 따로 매겨 "흐름단절 위치는 찾았는데 라벨만 다른가(type_confusion)"
  와 "위치 자체를 못 찾았나(순수 미탐)"를 분리한다. 위 흡수 케이스는 번복으로 위치를 잡았으니
  localization으론 커버 — 순수 미탐이 아니라 **라벨 귀속의 겹침 문제**임이 드러난다.
- **면접 Q&A 대비** — "F1 0.93이면 좋은 감지기인가?":
  ① 아니, **분해**해야 한다. 정밀도/재현율 중 어디서 깨지는지, 그 실패가 제품에서 지뢰(오탐)인지
     누수(미탐)인지. ② 미탐이면 **왜** 놓쳤나 — 위치를 못 찾았나(localization도 실패) vs 위치는
     찾고 라벨만 틀렸나(겹침·혼동). ③ 라벨이 상호배타가 아니면 "한 근거 다중 라벨"에서 모델이
     상위 서사로 흡수하는 구조적 미탐을 예상하고 채점을 그에 맞게 이중화한다.

**코드 위치**: `benchmarks/detection/measurements/README.md`(종합표·해석) ·
`benchmarks/detection/tests/test_measured_real.py`(동결 채점 회귀) ·
`score.py`의 localization 이중 채점 + `type_confusion` 분리.

**관련 노트**: `benchmarks/detection/README.md` 채점 방식(객체탐지식 매칭) · stage-1 CTER의
가짜모순/놓친모순 분리 철학(같은 "실패 방향" 사고를 STT층에 적용한 것).
