# troubleshooting.md — 함정 기록장

> **이 문서의 역할**: 작업 중 만난 함정(Trap)을 기록하고, **반복되면 재발 자체를 막는 층으로 승격**한다.
> 목표는 "같은 삽질 두 번 안 하기".
>
> - 앞으로 할 일은 [`plan.md`](plan.md), 완료 기록은 [`changeLog.md`](changeLog.md).
> - 여긴 **"이렇게 하지 마라 / 이 증상엔 이 해결"**(Trap). *"왜 이렇게 동작하는가"*(개념 이해)는 학습 노트로.

## T-### 번호 규칙과 승격 층

각 함정은 `T-###` ID를 갖는다. 번호는 사용자의 **글로벌 시퀀스를 잇는다**(마지막 알려진 값 = `T-026`, 이후 이 프로젝트 로컬 항목은 `T-027`부터). 함정은 세 층 중 하나에 산다:

| 층 | 위치 | 언제 |
|---|---|---|
| **프로젝트 로컬** | 이 파일 | 이 레포에서 처음 만난 함정 |
| **글로벌 원칙** | `~/.claude/CLAUDE.md` | 어느 레포에서든 재발하는 함정 → 원칙으로 승격 |
| **하드 가드(훅)** | `~/.claude/hooks/*.ps1` | 사람이 규칙을 알아도 실수하는 함정 → 자동 차단 |

**승격 기준**: 같은 함정을 **다른 맥락에서 2회 이상** 만나거나, 실수 비용이 크고 조용히 틀리는(무성 실패) 종류면 → 위층으로 올린다. 승격하면 이 파일엔 "→ 글로벌/훅으로 승격됨" 포인터만 남긴다.

## ✅ 이미 상위 층에서 처리된 것들 (여기서 재번호 안 함)

이 머신(Windows 11 + PowerShell 5.1 + Git Bash + 한국어)의 인코딩 함정들은 이미 **글로벌 CLAUDE.md**로 승격돼 있다 — 셋 다 "에러 없이 조용히 틀리는" 무성 실패라 선제 원칙화됨:

- **Git Bash grep/sed 멀티바이트 무성 실패** (C 로케일, 한글 패턴 0건 처리) → 텍스트 검색·치환은 ripgrep / PowerShell 리터럴. *(글로벌 「Windows 셸 한글·인코딩 원칙」 1)*
- **PowerShell 5.1 인라인 인자 한글 CP949 깨짐** (native exe에 넘기는 인라인 경로) → UTF-8 파일 경유(`-F`/stdin), 검증은 바이트(HEX) 대조. *(글로벌 원칙 2 / `T-026` = 한글 커밋 메시지 `.commit-msg-tmp`)*
- **파일 재생성 시 BOM·EOL 소실 → phantom diff** → 원본 BOM·EOL 감지 후 보존, 새 `.ps1`은 UTF-8 BOM 포함. *(글로벌 원칙 3)*

승격된 **하드 가드(훅)** 예:
- `block-main-edit.ps1` (PreToolUse) — main/master 브랜치에서 Write/Edit 차단 → "main 직접 작업 금지" 규칙의 자동화.
- `remind-korean.ps1` (UserPromptSubmit) — 매 턴 "사용자 대면 한국어" 지시 재주입.

## 프로젝트 로컬 함정

### T-027 · Windows 콘솔 cp949에서 Python `print`가 em-dash에 `UnicodeEncodeError`

- **증상**: `benchmarks/stt` 리포트 CLI를 Windows 콘솔에서 실행하면, `—`(em-dash)·`…` 같은 비-CP949 문자를 출력하는 순간 `UnicodeEncodeError: 'cp949' codec can't encode character` 로 죽는다. 정작 채점 로직은 멀쩡한데 **출력 단계**에서만 터진다.
- **원인**: Windows 기본 콘솔 코드페이지가 CP949라 Python `sys.stdout` 인코딩이 UTF-8이 아니다. 리포트 텍스트(한글 + 타이포그래피 기호)에 CP949 밖 문자가 섞이면 인코딩 실패.
- **해결**: CLI 진입점에서 stdout을 UTF-8로 재설정.
  ```python
  try:
      sys.stdout.reconfigure(encoding="utf-8")
  except (AttributeError, ValueError):
      pass  # 파이프/리다이렉트 등 reconfigure 불가 환경은 조용히 통과
  ```
- **재발 방지**: 사용자 대면 텍스트를 콘솔로 뿌리는 새 Python 진입점은 **처음부터** 이 재설정을 넣는다. (파일에 쓸 땐 `open(..., encoding="utf-8")`을 명시해 같은 함정 회피.)
- **관련**: 이건 글로벌 원칙 2(PowerShell→native exe 인라인 인자)와 **다른 갈래** — 저건 인자 전달, 이건 Python 프로세스 자신의 stdout. 파이썬 진입점이 늘어나 재발하면 글로벌 원칙으로 승격 고려.

### T-028 · 필드 setter 가드 비대칭 — 새 setter에 (빈값+중복) 가드 누락 시 무성 last-wins/데이터 손실

- **증상**: `synth.py` `_parse_fields`에서 `key=`는 헬퍼 `_set_key`가 (빈 값·중복) 둘 다 막는데, 나중에 추가한 `aliases=`/`canonical=`는 그 가드 없이 `x = val`로 **바로 대입**했다. 그래서 같은 필드를 두 번 주면 앞 값이 **조용히 소실**(last-wins)되고, 빈 값이면 surface로 **무성 fallback**한다. 골든이 에러 없이 틀림 — 예: `aliases=Lumi|aliases=루미에` → 골든에 `루미에`만 남아 채점기가 "Lumi" 전사를 **가짜 CTER 오류**로 집계. 테스트·검증 게이트는 통과하니 무성.
- **원인**: 기존 setter(`_set_key`)의 방어 규약을 **신규 필드 핸들러에 복제하지 않음**. "무성 실패 차단"이 이 빌더의 핵심 계약인데 새 필드만 계약 밖. 곁가지로, 무명 key 하위호환 판정을 `raw enumerate 인덱스 == 0`으로 해 선행 빈 필드가 있으면 유효 key를 'unknown field'로 **오거부**.
- **해결**: 새 명명 필드마다 `_set_key`와 **동일 패턴** — ① 빈 값 → 에러 ② 이미 설정됨(sentinel: `None`/`()`) → 중복 에러. 무명 위치 판정은 raw 인덱스 대신 **'첫 비어있지 않은 필드'**(`first` 플래그, 빈 필드 skip과 정합). *(`/code-review ultra` 2라운드, 6종 확정 — 상세 changeLog 2026-07-12.)*
- **재발 방지**: 파서/빌더에 "필드 → 상태" setter를 **추가**할 땐, 기존 setter의 (빈값+중복+위치) 가드를 그대로 미러링한다. 가드 없는 단순 대입(`x = val`)은 **무성 last-wins의 냄새** — 리뷰에서 이 패턴을 우선 의심.
- **관련**: 이 프로젝트 `stt_bench/synth.py` `_parse_fields`. 다른 파서/빌더에서 2회+ 재발하면 글로벌 원칙("신규 setter는 기존 setter 가드 미러링")으로 승격 검토. (개념 이해가 아니라 해결법이라 여기 Trap; 무성 실패 계열이라 선제 기록.)

### T-029 · 신뢰 불가 입력 필드를 산술/비교에 쓰면 한 레코드가 배치 전체를 죽인다 (격리 불변식 파괴)

- **증상**: `detect_bench` 감지 채점기에서, 예측(Claude 출력) statement의 `time_sec`을 grounding 힌트로 새로 쓰면서 `abs(seg.start_sec - time_sec)`를 계산. Claude는 `time_sec`을 숫자가 아니라 `"00:11"`/`"1:23"` 같은 **타임스탬프 문자열**로 흔히 낸다 → `float - str` **TypeError**가 미포착으로 올라가 그 flag 하나가 아니라 **채점 run 전체를 중단**. 같은 PR의 형제 기능(변형 type 라벨을 per-flag 강등해 "flag 하나가 run을 못 죽이게")이 세운 격리 불변식을, **새 힌트 필드가 다른 경로로 다시 깼다**.
- **원인**: 외부·신뢰 불가 소스(Claude JSON)의 필드를 **타입 검사 없이** 산술/비교에 투입. `if time_sec is not None` 가드는 "존재"만 볼 뿐 "숫자"는 안 본다 — 문자열은 통과해 산술에서 터진다. 힌트 도입 전엔 `time_sec`이 산술에 안 쓰여 이 경로가 없었으므로 **기능 추가가 크래시 경로를 신설**(회귀).
- **해결**: 경계에서 숫자만 통과시키는 강등 헬퍼 — `_num(x) = x if isinstance(x,(int,float)) and not isinstance(x,bool) else None`. 힌트 산술 진입점(`_pick`·`_span_grounding` 둘 다 — 크래시 사이트가 복수임에 유의)에서 `time_sec`·`start_sec`를 `_num`으로 강등하고, None이면 힌트를 무시해 **결정적 폴백**(첫 출현)으로 안전 진행. (적대적 리뷰 1R HIGH, 상세 changeLog 2026-07-12.)
- **재발 방지**: **신뢰 불가 입력의 필드를 산술·`abs`·`<`/`>`·인덱스에 쓸 땐, "존재(None 아님)"가 아니라 "타입(숫자/기대형)"을 검사**한다. `is not None` 가드는 타입 오염을 못 막는다. per-flag/per-record **격리 불변식**이 있는 파이프라인에 새 필드 소비를 추가할 때, 그 필드가 배치 전체를 죽일 수 있는지 항상 자문 — 특히 sibling 기능이 격리를 위해 존재한다면 새 코드가 그 격리를 우회하지 않는지 확인.
- **관련**: 이 프로젝트 `detect_bench/grounding.py` `_num`/`_pick`/`_span_grounding`. 다른 채점기/파이프라인에서 "신뢰 불가 필드 산술"로 2회+ 재발하면 글로벌 원칙으로 승격 검토. (해결법이라 Trap; "존재≠타입" 크래시 계열.)

### T-030 · isinstance 숫자 가드는 NaN을 못 막는다 — ==비반사성이 최근접/동등 필터를 무성 전멸시킴 (존재≠타입≠유한)

- **증상**: T-029에서 넣은 `_num(x) = x if isinstance(x,(int,float)) ... else None` 가드가 있는데도, 예측 `time_sec: NaN`(파이썬 `json.loads`는 bare `NaN` 리터럴을 **기본 허용**)이 가드를 통과. `_span_grounding`의 최근접 창 필터 `best = min(dist(...)); [c for c in cands if dist(c) == best]`에서 모든 `dist`가 `nan`이 되고 `nan == nan`은 **False**(비반사성) → 후보가 **전멸** → 유일하고 모호하지 않은 정당한 경계 창까지 grounding 거부 → 할루시 FP로 채점. 크래시가 아니라 **무성 오답**이라 T-029보다 악질.
- **원인**: NaN은 `isinstance(x, float)`가 True인 "타입은 숫자"다 — 타입 검사는 **유한성**을 보장하지 않는다. 게다가 NaN은 산술을 터뜨리지 않고(예외 없음) 비교 의미론만 조용히 오염시키므로, "산술에서 TypeError"를 막는 T-029식 가드로는 감지 자체가 안 된다.
- **해결**: `_num`에 유한성 검사 추가 — `and x == x` (NaN만 자기 자신과 불일치; `math.isnan` 없이 0-의존성 유지). 비정상 힌트는 계약대로 "무시하고 결정적 폴백"으로 강등된다. (적대적 리뷰 5R, 상세 changeLog 2026-07-13.)
- **재발 방지**: 신뢰 불가 숫자 필드의 가드는 3단 — **존재(None 아님) ≠ 타입(숫자) ≠ 유한(NaN/±inf)**. 특히 그 값이 `min`/`max`/`==` 필터·정렬 키·dict 키로 흘러가면 NaN 하나가 예외 없이 결과 집합을 통째로 비우거나 뒤섞는다. JSON 파서가 NaN을 허용하는지(파이썬 기본 허용) 반드시 확인.
- **관련**: 이 프로젝트 `detect_bench/grounding.py` `_num`. T-029(존재≠타입)의 직계 후속 — 같은 가드가 두 번 뚫렸으므로, 신뢰 불가 숫자 입력을 다루는 다른 프로젝트에서 1회라도 재발하면 글로벌 원칙("숫자 가드는 존재·타입·유한 3단") 승격 검토.

### T-031 · LLM 자유형식 출력에서 JSON을 '첫 파싱값'으로 뽑으면, 프롬프트가 담은 형식 예시를 모델이 에코할 때 강탈당한다

- **증상**: Claude 감지 응답에서 flags JSON을 추출하는 파서(`detect_bench/detect.py` `parse_detection_response`)가, 문자열을 훑어 **첫 번째로 파싱되는 JSON 값**을 답으로 잡았다. mock 픽스처(깔끔한 `{"flags":[...]}` 하나)에선 완벽했으나, 적대적 리뷰가 두 부류의 무성 오답을 드러냄: ⓐ 진짜 답 **앞에** 유효 JSON(빈 배열 `[]`, 산문 속 `[1,2]`, 내부 `statements` 배열)이 하나라도 있으면 그걸 답으로 잡음 — 특히 **빈 `[]` 하나가 "0건 감지"로 둔갑**해 recall을 조용히 오염. ⓑ 프롬프트가 형식·0건 예시(`{"flags": []}`, 예시 flag 오브젝트)를 담으므로 모델이 그 형식을 서두에 **되풀이(에코)**하면, "첫 `{"flags":...}` 컨테이너"를 잡는 방식조차 그 빈/예시 에코를 진짜 답 위로 강탈.
- **원인**: 자유형식 LLM 출력에서 **위치(첫 매칭)는 '진짜 답'의 신뢰 신호가 아니다**. 서문·형식 recap·코드펜스가 섞이고, 프롬프트가 자기 안에 정답 형식 예시를 담으면 그 예시가 출력에 새어 위치상 먼저 나올 수 있다. `all()` 게이트(bare 배열의 모든 원소가 flag스러워야 수용)도 원소 하나가 비정상이면 배열 전체를 버려 **부분손실**(이후 단일 원소만 주움)을 낳음.
- **해결**: 컨테이너를 **위치가 아니라 의미로** 고른다 — ⓐ 명시적 `{"flags": <list>}` 오브젝트를 최우선, 여럿이면 **실제 flag(dict) 수가 최대**인 것(동수면 마지막) → 0~1건짜리 형식 에코가 진짜 답을 못 이긴다. ⓑ flags 컨테이너가 하나도 없을 때만 차선(flag스러운 원소를 **하나라도** 가진 bare 배열 전체를 강등 경로로 — `all`이 아니라 `any` 게이트, 부분손실 방지). ⓒ **빈 목록은 유효한 0건, 추출 실패는 클린 에러**로 끝까지 구분(무성 [] 금지). 프롬프트 예시 좌표도 실제 골든 답과 일치시키지 말 것(에코 시 낙관 편향).
- **재발 방지**: **LLM이 낸 텍스트에서 구조를 뽑을 땐 "첫 번째로 파싱되는 것"을 절대 신뢰하지 말라.** 뽑으려는 구조의 **판별 키**(여기선 `flags` 키 + flag 원소 수)로 후보를 점수화해 고르고, "빈 유효 결과 vs 추출 실패"를 명시적으로 가르고, 프롬프트가 자기 출력 형식 예시를 담는지(그 예시가 에코될 수 있음) 자문한다. 다른 프로젝트에서 LLM-JSON 파싱으로 재발하면 글로벌 원칙 승격 검토. (해결법이라 Trap; "LLM 출력 구조 추출은 위치가 아니라 의미로" 계열. 적대적 리뷰 2R, 상세 changeLog 2026-07-13.)

### T-032 · 프롬프트 속 '파싱 가능한 형식 예시'는 언젠가 파서의 후보가 된다 — LLM-JSON 파서 휴리스틱은 극단 카디널리티(0건·1건·절단)에서 뒤집힌다

- **증상**: T-031 수정(의미 기반 선택: `flags` 키 + 최대 flag 수)이 '진짜 답 4건' 케이스에선 완벽했지만, 3R(xhigh) 검증에서 무성 오답 4변종이 전부 실측 재현: ⓐ 진짜 **0건**(`{"flags": []}`)이 프롬프트 예시 에코(1건짜리)에 패배(n=1>0 → 플레이스홀더 인용이 가짜 FP로 채점), ⓑ 진짜 **1건** + 후행 에코가 동수 타이브레이크(동수면 마지막)로 승리, ⓒ bare 배열 진짜 답 + 후행 규칙 에코 `{"flags": []}` → 래퍼 절대우선이라 **0건 둔갑**, ⓓ statement dict 3개짜리 기형 에코가 '임의 dict 카운트'로 진짜 2건을 수로 강탈. 별도로 **max_tokens 절단** 응답에선 완성된 첫 flag 조각만 fallback으로 인양돼 10건 감지가 '1건'으로 무성 축소(stop_reason 미검사와 결합).
- **원인**: ① 휴리스틱을 **'정답이 풍부한' 해피패스로만 회귀 테스트**했다 — 기존 테스트가 전부 진짜 답 4건 케이스라 0건·1건·동수·절단에서의 역전을 못 봤다. ② 근본적으로, **프롬프트가 스스로 파싱 가능한 예시**(`json.dumps` 예시 + verbatim `{"flags": []}` 리터럴)를 심는 한 에코는 언제나 파서의 유효 후보다 — 파서를 프롬프트 원문에 돌리면 더미 예시 flag가 추출될 정도(자기유발 문제). 파서 휴리스틱 보강은 하류 땜질.
- **해결**: ⓐ **뿌리** — 예시를 JSON 문법 밖 표기(`<...>` 플레이스홀더)로 렌더하고 0건 규칙에서도 파싱 가능한 리터럴 제거 → verbatim 에코가 **후보 자격 자체를 상실**. 회귀 불변식 테스트: `parse(build_detection_prompt(...))`는 반드시 ValueError. ⓑ **파서** — 카운트를 flag스러운 원소로 한정, 빈/무내용 컨테이너는 내용 있는 차선 후보에 양보, fallback은 '첫 후보'가 아니라 내용 최다, `"flags"` 텍스트는 있는데 유효 컨테이너 0이면 절단 의심 **클린 에러**(부분 조각 인양 금지). ⓒ **포트** — `stop_reason=max_tokens`를 클린 에러로 + `--max-tokens` 노브 배선. (적대적 리뷰 3R, 상세 changeLog 2026-07-13.)
- **재발 방지**: **LLM 출력 파서의 휴리스틱은 해피패스가 아니라 극단 카디널리티(0건·1건·동수·절단)로 검증하라.** 프롬프트에 출력 형식 예시를 넣을 땐 "이 예시가 그대로 에코되면 내 파서가 답으로 오인하는가?"를 자문하고, 오인 가능하면 예시를 **파싱 불가능한 표기**로 렌더하라 — 프롬프트와 파서는 한 몸으로 리뷰한다. T-031의 직계 후속(같은 계열 2회째) — 다른 프로젝트에서 재발하면 글로벌 원칙("LLM 형식 예시는 비파싱 표기 + 극단 카디널리티 테스트") 승격 검토.

### T-033 · 크레덴셜 게이트를 검증하는 테스트는 그 환경변수를 직접 지우고 시작해야 한다 — 키가 도착하는 순간 단위테스트가 과금 API를 때린다

- **증상**: `test_cli_claude_without_key_clean_error`가 "키 없음 → rc=2 클린 에러"를 검증하는데 `ANTHROPIC_API_KEY`를 지우지 않고 현재 환경을 상속 — 키 없는 개발 환경에선 영원히 초록이라 결함이 안 보인다. 이 프로젝트의 명시된 다음 단계가 '크레덴셜 도착'이라, 키를 export하는 순간 pytest가 골든 전사 전체를 **실제 api.anthropic.com에 POST**(과금, 타임아웃 도입 전이라면 행 가능)하고, 호출이 성공하면 rc=0이 되어 assert가 위양성으로 깨진다.
- **원인**: 테스트의 전제 조건('키 부재')을 테스트가 스스로 강제하지 않고 **환경의 우연**에 의존. '지금 내 환경에서 초록'은 '모든 환경에서 초록'이 아니며, 특히 **부재를 전제하는 테스트는 미래에 그 값이 생기는 순간 의미가 뒤집힌다**.
- **해결**: `monkeypatch.delenv("ANTHROPIC_API_KEY", raising=False)` 한 줄. (적대적 리뷰 3R 갭 스윕.)
- **재발 방지**: **환경변수 게이트(크레덴셜·피처 플래그·프록시)를 검증하는 테스트는 setenv/delenv로 전제를 명시 강제하라.** 크레덴셜 게이트 테스트는 "이 테스트가 키 있는 CI/동료 머신에서 돌면 무슨 일이 나는가"를 반드시 자문 — 네트워크·과금 부수효과가 있으면 그 자체로 결함이다.

### T-034 · 골든 하드케이스가 단일 신호(numeric time)로만 갈리면, 실측 LLM이 그 신호를 다른 포맷으로 낼 때 정탐이 무성으로 놓침(FN+FP)이 된다

- **증상**: 반복발화 하드케이스 골든에서 같은 문장이 디코이(s5)와 근거(s18)에 byte-동일·동일화자로 존재. 시프된 numeric `time_sec`(760)으로는 만점이지만, 실제 Claude 감지 경로에서 pred가 `time_sec`을 문자열(`"760"`)·시각(`"12:40"`)·단위표기(`"760초"`)로 내면 grounding이 조용히 디코이 s5로 붙어, 골든 f5(`{s18,s19}`)와 Jaccard 1/3<0.5로 **정탐이 놓침(FN)+가짜(FP)로 오채점**된다. 픽스처·골든·185테스트는 전부 초록이라 결함이 안 보였고, 적대적 리뷰가 스크립트 재현으로 잡았다.
- **원인**: 두 신호(speaker·time) 중 speaker가 반복발화에선 무력(디코이·근거 화자 동일)이라 **판별이 numeric time 하나에 전적으로 의존**. 그런데 예측 어댑터는 "신뢰 불가 예측을 per-flag 강등"하는 철학이라 비숫자 time을 조용히 버린다(`grounding._num`이 문자열·bool을 None으로) — 그 관용성이 하필 **유일한 신호**를 떨어뜨려, 관용성이 정확도를 깎는 역설이 된다. 프롬프트가 numeric 초를 요구해도 LLM은 포맷을 벗어날 수 있다.
- **해결**: 예측 경로(`labels._statement_from_data`의 `strict=False` 분기)에 `_coerce_pred_time` — 숫자문자열(`"760"`·`"760.0"`·`"760초"`)·시각(`"MM:SS"`·`"HH:MM:SS"`)을 초 float로 관용 파싱, 실패는 None(현행 안전 폴백 유지), 파싱된 ±inf/NaN은 `_is_num`으로 배제. **골든(strict)은 여전히 비숫자 time을 malformed로 거부**(엄격성 보존). 파라미터화 회귀 테스트로 `"760"`·`"12:40"`·`"760초"`가 만점 유지됨을 고정.
- **재발 방지**: **골든의 정답이 실측 생산자(LLM)가 내는 힌트 하나에 전적으로 의존하면, 그 힌트의 ingest를 관용화하라** — 또는 2차 disambiguation을 골든에 심어 단일신호 의존을 없애라. 하드케이스를 설계할 때 "이 판별 신호가 실측 포맷 변이(문자열·시각·단위)에서도 살아남는가"를 자문. mock 픽스처가 초록인 것은 "실측에서도 초록"을 뜻하지 않는다(mock에선 안 밟히는 실측 전제 = PR #10 계열 패턴).
